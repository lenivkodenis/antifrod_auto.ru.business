# Архитектура Telegram-бота на базе API Авто.ру Бизнес

## Вводная

У Авто.ру Бизнес есть API-интеграции. По открытой документации доступны несколько API-контуров, включая DMS.API, Inspections.API и Appraisal.API. Доступ к спецификациям методов находится в Swagger и требует авторизации в Авто.ру Бизнес.

## Предварительный вывод

Технически идея Telegram-бота, который получает данные из Авто.ру Бизнес по API и формирует отчет собственнику, выглядит реализуемой.

Но есть важное ограничение: нужно проверить, какие именно поля доступны через API:

- оценки;
- статус оценки;
- статус мобильного осмотра;
- цена собственнику;
- маржа;
- цена продажи;
- ОЦП;
- процент цены к рынку;
- GM2;
- статус согласования;
- попадание автомобиля на склад;
- данные актуального склада;
- публикация;
- звонки/чаты;
- продвижение;
- факт продажи.

Пока эти поля не сверены со Swagger, нельзя утверждать, что весь отчет можно собрать напрямую по API без Excel-выгрузок.

## Базовая архитектура

```text
Авто.ру Бизнес API
        ↓
Интеграционный слой / ETL
        ↓
База данных проекта
        ↓
Модуль расчета риск-сигналов
        ↓
Генератор отчетов
        ↓
Telegram-бот / HTML-отчет / PDF-отчет
```

## Компоненты

### 1. API-коннектор Авто.ру Бизнес

Задачи:

- получить OAuth-токен;
- обращаться к нужным API;
- забирать оценки, осмотры, склад и продажи;
- обновлять данные по расписанию;
- логировать ошибки интеграции.

### 2. Слой нормализации данных

Задачи:

- привести поля API к внутренней схеме;
- связать оценки, склад и продажи;
- схлопнуть дубли по VIN или внутреннему ID автомобиля;
- убрать чувствительные данные там, где они не нужны для отчета;
- подготовить агрегаты для собственника.

### 3. Модуль правил

Задачи:

- проверить экономику на ручной расчет;
- проверить мобильный осмотр;
- проверить завершенность оценки;
- проверить согласование;
- проверить маржу ниже порога;
- проверить выкуп выше ОЦП;
- проверить цену продажи относительно рынка;
- позже добавить сток и выход.

### 4. Хранилище данных

Минимальный MVP может начинаться с простой базы:

- PostgreSQL;
- SQLite для прототипа;
- Google Sheets / CSV только для раннего теста;
- позже — полноценное backend-хранилище.

Хранить нужно не только текущий срез, но и историю, чтобы видеть динамику.

### 5. Генератор отчетов

Форматы:

- короткая Telegram-сводка;
- HTML-отчет;
- PDF-отчет;
- JSON для будущего интерфейса.

### 6. Telegram-бот

Функции:

- авторизация пользователя;
- выбор периода;
- выбор дилерского центра;
- запрос отчета;
- показ короткой сводки;
- выдача ссылки на HTML;
- отправка PDF;
- показ списка критичных сигналов.

## Возможный сценарий пользователя

1. Собственник пишет боту: `Отчет вход`.
2. Бот спрашивает период: `Сегодня / Неделя / Месяц / Свой период`.
3. Бот формирует короткую сводку:
   - принято автомобилей;
   - нарушений процесса;
   - экономико-рыночных рисков;
   - критичных сигналов.
4. Бот предлагает кнопки:
   - `Открыть HTML-отчет`;
   - `Скачать PDF`;
   - `Показать критичные`;
   - `По сотрудникам`;
   - `По источникам`.

## Два возможных режима интеграции

### Режим 1. MVP через Excel

Подходит для быстрой проверки продукта.

- дилер выгружает Excel;
- файл загружается в систему;
- скрипт считает сигналы;
- бот отдает отчет.

Плюсы:

- быстро запустить;
- не нужно сразу разбираться с API;
- можно проверить ценность продукта.

Минусы:

- ручная выгрузка;
- нет онлайн-актуальности;
- сложнее масштабировать.

### Режим 2. API-интеграция

Подходит для продукта.

- система регулярно забирает данные по API;
- сигналы пересчитываются автоматически;
- собственник вызывает отчет в Telegram;
- можно делать ежедневные/еженедельные уведомления.

Плюсы:

- автоматизация;
- регулярность;
- масштабируемость;
- ближе к продукту Авто.ру Бизнес.

Минусы:

- нужно получить доступы;
- нужно изучить Swagger;
- нужно проверить доступность нужных полей;
- нужна безопасная архитектура.

## Что нужно проверить в Swagger

Для блока входа автомобиля нужно найти методы и поля, которые позволяют получить:

- список оценок за период;
- статус оценки;
- ID оценки;
- ID автомобиля на складе;
- статус `На складе`;
- дату создания и обновления;
- мобильный осмотр и его статус;
- тип контракта;
- сотрудника / оценщика в обезличенной или разрешенной форме;
- цену собственнику;
- подготовку;
- маржу в рублях и процентах;
- цену продажи;
- процент цены к рынку;
- ОЦП;
- прогноз минимальной цены;
- прогноз срока продажи;
- согласованную цену;
- кто согласовал;
- GM2, если доступна.

## Безопасность

Telegram не должен быть местом хранения чувствительных данных.

Принципы:

- в Telegram показывать агрегаты;
- детальные отчеты открывать по защищенной ссылке;
- использовать авторизацию пользователей;
- не отправлять VIN, ФИО, телефоны в открытых сообщениях;
- ограничивать срок жизни ссылок;
- логировать запросы отчетов;
- хранить токены API отдельно от кода.

## Рекомендованный путь

1. Продолжить MVP на Excel, чтобы быстро проверить продуктовую ценность.
2. Параллельно запросить доступ к Swagger и API-реквизиты.
3. Сверить поля Excel с полями API.
4. Собрать таблицу `поле отчета → поле API → доступно/не доступно`.
5. После этого принимать решение, можно ли полностью перейти с Excel на API.

## Главный вывод

Бот через API возможен концептуально. Но ключевая развилка не в Telegram, а в доступности данных:

> Если API отдает оценки, осмотры, склад, экономику и продажи в нужной детализации, бот может строить отчет автоматически. Если часть полей недоступна, потребуется гибрид: API + Excel/внутренние выгрузки дилера.
