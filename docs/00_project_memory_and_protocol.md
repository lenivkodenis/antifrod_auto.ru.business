# Память проекта и NDA-протокол

Этот файл фиксирует пройденный путь проекта так, чтобы к нему можно было вернуться в новом чате, в Codex или при продолжении работы над продуктом.

## NDA-принцип

Репозиторий используется как методологическая и продуктовая память, а не как хранилище клиентских данных.

### Нельзя сохранять в репозиторий

- сырые Excel-выгрузки клиентов;
- реальные VIN;
- ФИО сотрудников;
- телефоны, email, персональные данные;
- коммерческие условия конкретных сделок;
- реальные цены сделок в связке с автомобилем, VIN, сотрудником или клиентом;
- скриншоты с идентификаторами клиента, сотрудника, сделки или автомобиля;
- cookies, токены, пароли, ключи доступа;
- любые материалы, по которым можно восстановить конкретного дилера или конкретную сделку без разрешения.

### Можно сохранять в репозиторий

- методологию;
- продуктовые гипотезы;
- обезличенные выводы;
- правила контроля;
- схемы данных без чувствительных значений;
- синтетические примеры;
- промпты;
- инструкции для агента;
- код анализа;
- структуру будущих отчетов;
- обобщенные паттерны рисков.

## Главная цель проекта

Создать систему контроля качества сделок с автомобилями с пробегом для дилеров в России. Система должна быть применима к логике Авто.ру Бизнес и потенциально может стать:

- консалтинговым продуктом;
- методологией аудита;
- регулярным управленческим отчетом;
- будущим продуктовым слоем / модулем / плагином Авто.ру Бизнес.

## Целевая аудитория

- собственник дилерского бизнеса;
- генеральный директор;
- руководитель направления автомобилей с пробегом;
- руководитель отдела продаж;
- внутренний контролер / аудитор.

## Ключевая идея

Система должна показывать не список всех автомобилей, а список управленческих отклонений.

Не обвинение сотрудника, а сигнал:

> Сделка прошла с отклонением от нормального процесса и требует проверки.

## Эволюция рамки проекта

### Первичная идея

Проект начинался как система контроля качества выкупа автомобилей с пробегом.

### Расширение рамки

Затем рамка была расширена: контролировать нужно не только выкуп, а все основные типы сделок с автомобилями с пробегом:

- trade-in;
- trade-up;
- выкуп;
- комиссия;
- ком-трейд.

### Текущая рамка

Риски не раскладываются по каждому типу договора, потому что большинство ошибок у байера, руководителя и продавца повторяются во всех типах контрактов.

Основная структура проекта — три блока жизненного цикла автомобиля:

1. **Вход автомобиля** — оценка, мобильный осмотр, экономика, согласование, попадание на склад.
2. **Управление автомобилем на стоке** — публикация, продвижение, цена, переоценка, спрос, срок хранения, GM2.
3. **Выход / продажа** — скидка, продажа ниже рынка, продажа без публикации, итоговая маржа, отличие факта от плана.

## Текущий фокус

Сейчас проект находится на первом большом блоке:

# Контроль входа автомобиля

Ключевые темы:

- завершенность мобильного осмотра;
- завершенность оценки до попадания автомобиля на склад;
- наличие признаков индивидуального экономического расчета;
- проверка дефолтной / шаблонной экономики;
- настройка срока продажи;
- прогноз цены продажи;
- анализ GM2, если поле есть в выгрузке;
- управленческий ценовой коридор;
- сигналы для собственника.

## Важные методологические решения

### 1. Диагностика не является must-have в Авто.ру Бизнес

Диагностика часто живет во внутренних системах дилера и подгружается позже как файл автомобиля.

Поэтому отсутствие диагностики в выгрузке Авто.ру Бизнес нельзя трактовать как факт отсутствия диагностики.

### 2. Мобильный осмотр — ключевой цифровой след

Для контроля входа важнее смотреть на завершенность мобильного осмотра.

Формулировка:

> Автомобиль попал на склад без завершенного мобильного осмотра в Авто.ру Бизнес. Это не доказывает, что его физически не смотрели, но в системе нет завершенного цифрового следа осмотра.

### 3. Экономику нельзя трактовать жестко без истории изменений

Авто.ру Бизнес может предлагать расчет экономики по умолчанию. Если оценщик не менял параметры, дефолтные значения могут остаться в оценке.

Без истории изменений нельзя утверждать:

> Байер ничего не считал.

Корректная формулировка:

> Экономика оценки выглядит шаблонной. Не видно признаков индивидуальной настройки расчета.

### 4. GM2 важнее простой плановой маржи

Если в выгрузке есть GM2, его нужно анализировать обязательно:

- GM2 в рублях;
- GM2 в процентах;
- GM2 ниже целевого порога;
- случаи, где GM1 выглядит приемлемо, но GM2 слабая или отрицательная.

## Уже зафиксированные первые сигналы входа

- автомобиль на складе, но оценка не завершена;
- автомобиль на складе без завершенного мобильного осмотра;
- экономика оценки выглядит шаблонной;
- не задан срок продажи;
- не настроена прогнозная цена продажи;
- нет управленческого ценового коридора;
- маржа ниже целевого порога;
- GM2 ниже целевого порога;
- повторный VIN требует связывания оценок в историю автомобиля.

## Принцип сохранения прогресса

В репозиторий нужно регулярно сохранять:

- новые методологические решения;
- новые правила контроля;
- новые продуктовые гипотезы;
- результаты анализа в обезличенном виде;
- промпты для продолжения работы;
- структуру будущих отчетов;
- инструкции для Codex/агента.

Не нужно сохранять каждую промежуточную мысль. Нужно сохранять только то, что помогает восстановить проект и продолжить работу без потери контекста.
